# Chrome拡張機能の「強力な権限」とUXのトレードオフについて

**VoiceMux Bridge** は、スマートフォンの音声入力をPCのあらゆるウェブサイトへシームレスに転送するツールです。インストール時に表示される「すべてのウェブサイト上の全データの読み取りと変更」という警告の意図と、プライバシーへの配慮について開発チームの見解をまとめました。

---

## 1. なぜ「信頼されていません」という警告が出るのか？

### 背景
Chrome ウェブストアからのインストール時に以下のメッセージが表示されることがあります。

> **警告:** この拡張機能は、セーフ ブラウジング保護強化機能で信頼されていません。

### 理由
これはプログラムの欠陥ではなく、Google Chrome のセキュリティ評価アルゴリズムによるものです。主に以下の条件で表示されます。

- **開発者実績の蓄積期間:** デベロッパー登録からの期間が短い。
- **ユーザー数:** 公開直後で利用者が限定的である。

**VoiceMux の対応:**
私たちはすべてのソースコードを GitHub で公開しており、誰でも安全性を監査できる状態にしています。利用者が増えるにつれて、この警告は自動的に表示されなくなります。

---

## 2. なぜ「全サイトへのアクセス権限」が必要なのか？

### 理由：シームレスな「ハンズフリー体験」の実現
VoiceMux Bridge の核心は、**「PC側に触れることなく、スマホに話しかけるだけで文字が入力される」**という体験です。

この「魔法のような体験」を実現するためには、拡張機能が以下の動作を行う必要があります。

1. **待機:** ユーザーがどのタブ（Gemini, ChatGPT, Gmail等）を開いても、音声データを受け取れる状態にする。
2. **自動検知:** タブを切り替えた際、新しいページ上の入力フィールドを即座に特定して文字を流し込む。

もし権限を「クリックした時だけ」に制限した場合、**タブを切り替えるたびに毎回拡張機能のアイコンをクリックして許可を与える**操作が必要になります。これでは音声入力の利便性が損なわれ、ツールの存在意義が失われてしまいます。

---

## 3. プライバシーと安全性について

私たちは、強力な権限を要求する責任として、以下の設計と情報公開を徹底しています。

- **クライアント側のオープンソース化:** 拡張機能（クライアント）のソースコードはすべて GitHub で公開しています。これにより、「ブラウザから送信される前にデータが正しく暗号化されているか」「入力テキスト以外のデータを取得していないか」を誰でも監査可能です。
- **エンドツーエンド暗号化 (E2EE):** 音声データやテキストは、ブラウザとスマホの間で直接暗号化されます。サーバー側はソースコード非公開のプライベート環境で運用されていますが、クライアント側で暗号化が完結しているため、サーバー（当サイト）でさえ中身を読み取ることは物理的に不可能です。
- **サーバーに鍵を渡さない仕組み:** スマホとのペアリングに使用する暗号鍵は、URLのハッシュフラグメント（`#` 以降の部分）として共有されます。ブラウザの仕様上、この部分はサーバーに送信されないため、鍵の情報がネットワークを流れることはありません。
- **最小限のデータ収集:** 拡張機能は設定（Room ID / Key）をローカルに保存するのみで、閲覧履歴や個人情報を外部へ送信することはありません。

---

## 4. 他のツールとの比較

多くのインフラ系ツール（1Password, Grammarly, DeepL等）も、同様の理由で「全サイトへのアクセス権限」を要求します。「いつユーザーが入力を必要とするか予測できない」ツールにとって、この権限は利便性と安全性を両立するための標準的な選択です。

---

## 結論

VoiceMux Bridge は、**「究極の入力体験」を提供するために現在の権限設定を選択しました。** 警告は強力に見えますが、私たちは透明性と技術（E2EE）によってその信頼に応えます。
